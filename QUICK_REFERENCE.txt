TRANSPORT MECHANISM
══════════════════════════════════════════════════════════════════════════════════

Platform-dependent serialization:

Unix/macOS:
  1. JSON.stringify({ mcpServers })
  2. Pass JSON directly as string: --mcp-config '{"mcpServers":{...}}'

Windows:
  1. JSON.stringify({ mcpServers })
  2. Write to temp file: /tmp/mcp-config-<pid>-<timestamp>.json
  3. Pass file path: --mcp-config /tmp/mcp-config-...json
  4. Cleanup: Delete temp file after spawn

DYNAMIC VS STATIC
══════════════════════════════════════════════════════════════════════════════════

STATIC (.mcp.json or `claude mcp add`)
├─ Config stored on disk
├─ Read at Claude startup
├─ URLs hardcoded
└─ Fixed set of servers

YOHO-REMOTE (DYNAMIC)
├─ Config assembled at runtime
├─ Session-specific URLs (from startYohoRemoteServer)
├─ API-sourced project paths (from getYohoAuxMcpServers)
├─ Org isolation headers added automatically
├─ Per-session tool restrictions (allowedTools/disallowedTools)
└─ Runtime fallbacks (HTTP vs stdio)

BOTH USE: Same --mcp-config CLI mechanism
